## 10.6 Evaluation Metrics — Classification

---

### 📖 Definition
> **Evaluation metrics** measure the quality of a classifier's predictions. Because accuracy alone is insufficient (especially on imbalanced data), we use a suite of metrics — each focusing on a different aspect of what it means to predict correctly.

---

## 1. Accuracy

### 📖 Definition
> **Accuracy** = the fraction of total predictions that were correct.

```
Accuracy = Correct Predictions / Total Predictions
```

### 🧑‍🏫 Layman Explanation
If you take a 10-question quiz and get 8 right, your accuracy is 80%. Simple and intuitive — but it breaks down completely when one class is much rarer than the other.

### ⚙️ The Fatal Flaw — Imbalanced Data
```
100 emails: 95 Not Spam | 5 Spam
Model strategy: always predict "Not Spam"

Accuracy = 95/100 = 95% ✅ (looks great!)
Spam correctly detected = 0/5 ❌ (completely useless)
```

### ⭐ Key Highlights
> 🔑 `Accuracy = TP + TN / (TP + TN + FP + FN)`
> 🔑 **Only reliable when classes are balanced**
> 🔑 ⚠️ High accuracy on imbalanced data is a red flag, not a success

---

## 2. Confusion Matrix

### 📖 Definition
> A **Confusion Matrix** is a 2×2 table (for binary classification) showing how predictions compare to actual labels across all four possible outcomes.

### 🧑‍🏫 Layman Explanation
It's a report card that breaks down all four types of outcomes in one place. Not just "how many did you get right?" but specifically: where did you go right, and where did you go wrong?

### ⚙️ Technical Explanation

```
                    PREDICTED
                  Positive   Negative
ACTUAL  Positive |   TP    |   FN   |   ← Missed positives (dangerous!)
        Negative |   FP    |   TN   |   ← False alarms
```

**The Four Cells — Memorize These:**

| Term   | Full Name      | Meaning                        | In Plain English |
|--------|----------------|--------------------------------|---------------------------------------|
| **TP** | True Positive | Predicted Positive, IS Positive | Correctly caught spam |
| **FP** | False Positive | Predicted Positive, NOT Positive | Good email wrongly flagged as spam |
| **TN** | True Negative | Predicted Negative, IS Negative | Correctly let good email through |
| **FN** | False Negative | Predicted Negative, NOT Negative | Spam that slipped through |

**From the PDF example (Spam case):**
```
                  Predicted Spam   Predicted Not-Spam
Actual Spam     |     600 (TP)    |    300 (FN)       |
Actual Not-Spam |  100 (FP)    |   9000 (TN)       |
```

> 🔑 **Every metric (Precision, Recall, F1, Accuracy) is derived from TP, FP, TN, FN**

### 💡 Code

```python
from sklearn.metrics import confusion_matrix, ConfusionMatrixDisplay
import matplotlib.pyplot as plt

cm = confusion_matrix(y_test, y_pred)
print(cm)
# [[TN, FP],
#  [FN, TP]]

disp = ConfusionMatrixDisplay(confusion_matrix=cm, display_labels=["Not Spam", "Spam"])
disp.plot(cmap='Blues')
plt.title("Confusion Matrix")
plt.show()
```

## 3. Precision

### 📖 Definition
> **Precision** = of all the cases the model predicted as Positive, how many were actually Positive?

```
Precision = TP / (TP + FP)
```

### 🧑‍🏫 Layman Explanation
If your security system sends 10 alerts and 7 are real threats, your precision is 70%. Precision asks: **"When you raise the alarm, are you usually right?"** Low precision = crying wolf too often.

### ⚙️ When to Use
Use Precision when **False Positives are costly**:
- Spam filter: blocking a legitimate email is annoying
- Fraud alert: freezing a valid customer's account is bad
- Legal evidence: wrongly accusing an innocent person

```python
from sklearn.metrics import precision_score

precision = precision_score(y_test, y_pred)
print(f"Precision: {precision:.2%}")
# Precision: 70.00% → When model says "Spam", it's right 70% of the time
```

### ⭐ Key Highlights
> 🔑 Precision = **quality of positive predictions**
> 🔑 High precision → **few false alarms**
> 🔑 Improving precision usually hurts recall (fewer positives predicted overall)

---

## 4. Recall (Sensitivity / True Positive Rate)

### 📖 Definition
> **Recall** = of all the actual Positive cases that exist, how many did the model successfully find?

```
Recall = TP / (TP + FN)
```

### 🧑‍🏫 Layman Explanation
If 10 people actually have a disease and your test catches 7 of them, your recall is 70%. Recall asks: **"Out of all the real positives, how many did you catch?"** Low recall = you're missing things that matter.

### ⚙️ When to Use
Use Recall when **False Negatives are costly**:
- Disease detection: missing a sick patient is dangerous
- Security systems: missing a real threat is catastrophic
- Fraud detection (when every fraud missed = real loss)

```python
from sklearn.metrics import recall_score

recall = recall_score(y_test, y_pred)
print(f"Recall: {recall:.2%}")
# Recall: 70.00% → Model catches 70% of all actual spam
```

### ⭐ Key Highlights
> 🔑 Recall = **coverage of actual positives**
> 🔑 High recall → **few missed positives (FN)**
> 🔑 Recall also called **Sensitivity** or **True Positive Rate (TPR)**

---

## 5. F1 Score

### 📖 Definition
> **F1 Score** = the **harmonic mean** of Precision and Recall. It balances both — a high F1 requires both to be good simultaneously.

```
F1 = 2 × (Precision × Recall) / (Precision + Recall)
```

### 🧑‍🏫 Layman Explanation
Precision and Recall pull in opposite directions: increasing one usually decreases the other. F1 is like a referee that says: "You only get a high score if you're good at BOTH." If Precision = 100% but Recall = 0%, F1 = 0. You can't game it by focusing on just one.

### ⚙️ Why Harmonic Mean (not average)?
Simple average of 90% and 10% = 50% (misleadingly high).
Harmonic mean: `2 × (0.9 × 0.1) / (0.9 + 0.1) = 0.18` → correctly low. It **punishes extreme imbalance** between the two.

```python
from sklearn.metrics import f1_score, classification_report

f1 = f1_score(y_test, y_pred)
print(f"F1 Score: {f1:.2%}")

# Full report: Precision, Recall, F1 per class
print(classification_report(y_test, y_pred, target_names=["Not Spam", "Spam"]))
```

**F1 variants for multi-class:**
```python
# macro: average F1 across all classes (treats each class equally)
f1_macro = f1_score(y_test, y_pred, average='macro')

# weighted: weighted by class support (number of actual instances)
f1_weighted = f1_score(y_test, y_pred, average='weighted')

# Your Jindal Steel project used F1-macro for multiclass fault classification ← remember this
```

### ⭐ Key Highlights
> 🔑 F1 = **balance between Precision and Recall**
> 🔑 Best metric for **imbalanced datasets** (more honest than accuracy)
> 🔑 Use **F1-macro** for multi-class when all classes matter equally

## 6. ROC-AUC

### 📖 Definition
> **ROC (Receiver Operating Characteristic)** curve plots **True Positive Rate (Recall) vs False Positive Rate** across all possible decision thresholds. **AUC (Area Under the Curve)** summarizes the entire curve as a single score.

```
TPR (True Positive Rate)  = TP / (TP + FN)   ← same as Recall
FPR (False Positive Rate) = FP / (FP + TN)   ← how often we falsely alarm
```

### 🧑‍🏫 Layman Explanation
Think of AUC as answering: **"If I randomly pick one real positive and one real negative, how likely is my model to rank the positive higher?"** AUC = 1.0 → perfect; AUC = 0.5 → no better than random guessing (diagonal line). It tells you about the model's **overall discriminating power** regardless of which threshold you choose.

### ⚙️ Technical Explanation

```
ROC Curve shape:

TPR
1.0 |  .(perfect classifier)
    | /  \
    |/    · (good model — hugs top-left)
    |   ·
    | ·
    |·--------  (random = diagonal)
    +----------→ FPR
   0.0          1.0

AUC interpretation:
  AUC = 1.0   → Perfect classifier
  AUC = 0.9+  → Excellent
  AUC = 0.8-0.9 → Good
  AUC = 0.7-0.8 → Fair
  AUC = 0.5   → Random (useless)
  AUC < 0.5   → Worse than random
```

```python
from sklearn.metrics import roc_auc_score, roc_curve
import matplotlib.pyplot as plt

# Need probability scores (not just class labels)
y_proba = model.predict_proba(X_test)[:, 1]   # P(Class = 1)

auc = roc_auc_score(y_test, y_proba)
print(f"AUC: {auc:.4f}")

# Plot ROC Curve
fpr, tpr, thresholds = roc_curve(y_test, y_proba)
plt.plot(fpr, tpr, label=f"Model (AUC = {auc:.2f})")
plt.plot([0, 1], [0, 1], 'k--', label="Random (AUC = 0.50)")
plt.xlabel("False Positive Rate")
plt.ylabel("True Positive Rate (Recall)")
plt.title("ROC Curve")
plt.legend()
plt.show()
```

### ⭐ Key Highlights
> 🔑 AUC answers: **"How well can the model separate classes overall?"**
> 🔑 **Threshold-independent** — evaluates all thresholds at once
> 🔑 Use when you want to compare models without committing to a threshold
> 🔑 Works with `predict_proba()` — needs probability scores, not just labels

## ⚡ Metrics Quick-Reference Card

| Metric        | Formula                 | Avoids          | Use When |
|---------------|-------------------------|-----------------|----------------------|
| **Accuracy**  | (TP+TN) / Total         | —               | Balanced classes only |
| **Precision** | TP / (TP+FP)            | False Positives | FP is costly (spam, fraud alerts) |
| **Recall**.   | TP / (TP+FN)            | False Negatives | FN is costly (disease, security) |
| **F1 Score**  | 2×P×R / (P+R)           | Both FP & FN    | Imbalanced data, need balance |
| **AUC-ROC**   | Area under TPR/FPR curve| —               | Compare models, threshold-free |

## 💡 The Precision-Recall Tradeoff — Visualized

The core difference
They answer two different questions about your "Predicted 1" and "Actual 1" groups:

**Precision** = "Of everything I predicted as positive, how much did I get right?"
 → TP / (TP + FP). 
 Precision is about trustworthiness of your positive predictions.

**Recall** = "Of everything that's actually positive, how much did I catch?" 
→ TP / (TP + FN). 
Recall is about coverage — did you miss any?

Using your matrix: precision (60.9%) asks "when the model flags a plate as class 1, can I trust that?" 
Recall (51.9%) asks "of all the plates that are actually class 1, how many did the model actually find?"
Why they trade off
They pull in opposite directions because of where the model draws its decision threshold.

Make the model more "**cautious**" about predicting positive (raise the threshold) → fewer false positives → precision goes up, but you also catch fewer true positives → recall goes down.

Make the model more "**aggressive**" about predicting positive (lower the threshold) → you catch more true positives → recall goes up, but you also flag more things wrongly → precision goes down.

```
Lower threshold (e.g. 0.3):
  → More things flagged as Positive
  → Recall ↑ (catch more real positives)
  → Precision ↓ (more false alarms)

Higher threshold (e.g. 0.7):
  → Fewer things flagged as Positive
  → Precision ↑ (when you flag, you're usually right)
  → Recall ↓ (miss more real positives)

                HIGH RECALL ←──── Threshold ────→ HIGH PRECISION
               (catch everything)           (only flag when sure)
```

```python
# See how threshold affects Precision & Recall
from sklearn.metrics import precision_recall_curve

precisions, recalls, thresholds = precision_recall_curve(y_test, y_proba)

# Plot Precision-Recall curve
plt.plot(recalls, precisions)
plt.xlabel("Recall")
plt.ylabel("Precision")
plt.title("Precision-Recall Tradeoff")
plt.show()
```
