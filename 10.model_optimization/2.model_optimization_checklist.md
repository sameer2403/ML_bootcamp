## 13.5 Model Optimization Checklist

### Definition
A systematic, ordered checklist to improve model performance — covering data quality, feature handling, model-level tuning, and validation — rather than randomly tweaking things.

### Layman Explanation
It's like servicing a car when it's not running well: you don't start by replacing the engine (the model). You first check the fuel (data quality), then the tires and filters (features), *then* tune the engine (model optimization), and finally take it for a test drive to confirm it actually runs better (validation).

### Technical Explanation

**Core Principles**
- Improve performance **step by step** (systematically, not randomly)
- **Fix data before fixing the model** — a great model can't compensate for bad data
- **Optimize systematically** — follow an order, don't jump straight to complex model changes

**Step 1 — Data Comes First**
| Action              | Details |
|---------------------|---------------------|
| Data quality checks | Handle missing values, outliers, duplicates |
| Train-test split early | Use **stratified** split for classification (keeps class ratio) |
| Avoid data leakage | Isolate the test set completely during all processing (ties to 12.6) |

**Step 2 — Feature Optimization**
| Action              | Details |
|---------------------|---------------------|
| Feature Engineering | Create/transform features to expose patterns to the model |
| Feature Scaling     | Scale using **training data statistics only** — keeps train/test distribution consistent, avoids leakage |
| Feature Selection   | Remove irrelevant or redundant features to reduce noise |

**Step 3 — Model Optimization**
| Action                 | Details                 |
|------------------------|------------------------|
| Handle class imbalance | Balance class distribution (e.g., SMOTE, class weights) |
| Model selection        | Compare multiple algorithms, not just one |
| Hyperparameter tuning  | Grid Search or Randomized Search (from 13.1) |
| Regularization         | Control model complexity (prevents overfitting) |

**Step 4 — Validation & Evaluation**
| Action           | Details           |
|------------------|------------------|
| Cross Validation | Use CV for a reliable performance estimate (ties to 12.4) |
| Evaluation Metric | Choose correctly: Accuracy vs Precision/Recall/F1/ROC-AUC (ties to 12.6) |
| Error Analysis | Deeply analyze wrong predictions to understand *where* and *why* the model fails |

### Example
A classification model has 65% recall on the minority class. Following the checklist in order:
1. Check data → find 8% missing values in a key feature → impute properly.
2. Check features → scale numeric features using train-only stats.
3. Check model → apply `class_weight="balanced"`, then tune `max_depth`/`n_estimators` via Randomized Search.
4. Validate → run 5-fold Stratified CV, check F1 instead of accuracy, and manually inspect the false negatives to see what they have in common.

### Highlights 🔑
- **Order matters**: Data → Features → Model → Validation. Jumping straight to hyperparameter tuning while ignoring bad data wastes time.
- Feature scaling must be fit on **train only** — a recurring theme tying back to data leakage (12.6).
- Error analysis is the step most people skip — but it's how you discover *systematic* weaknesses (e.g., model always fails on a specific subgroup).

### Brain Map 🧠
```
MODEL OPTIMIZATION CHECKLIST  (do in this order)
   1. DATA         → quality checks, early stratified split, no leakage
   2. FEATURES     → engineer, scale (train-only), select
   3. MODEL        → imbalance handling, model selection, tuning, regularization
   4. VALIDATE     → cross-validation, right metric, error analysis

Mnemonic: "D-F-M-V — Dirty Floors Make Vacuuming necessary."
(Data, Features, Model, Validation — clean each layer before moving to the next.)
```

### Code
```python
from sklearn.model_selection import train_test_split, StratifiedKFold, RandomizedSearchCV
from sklearn.pipeline import Pipeline
from sklearn.preprocessing import StandardScaler
from sklearn.feature_selection import SelectKBest, f_classif
from sklearn.ensemble import RandomForestClassifier
from sklearn.metrics import classification_report, confusion_matrix

# --- Step 1: Data — stratified split early, before any processing ---
X_train, X_test, y_train, y_test = train_test_split(
    X, y, test_size=0.2, stratify=y, random_state=42
)
# (handle missing values / outliers / duplicates here, fit only on X_train)

# --- Step 2 & 3: Features + Model, chained in a Pipeline (prevents leakage) ---
pipe = Pipeline([
    ("scaler", StandardScaler()),                 # fit on train only
    ("select", SelectKBest(f_classif, k=10)),      # feature selection
    ("model", RandomForestClassifier(
        class_weight="balanced", random_state=42   # handle class imbalance
    ))
])

param_dist = {
    "model__n_estimators": [100, 200, 300],
    "model__max_depth": [3, 5, 7, None]
}

search = RandomizedSearchCV(
    pipe, param_distributions=param_dist, n_iter=8,
    cv=StratifiedKFold(n_splits=5), scoring="f1", random_state=42
)
search.fit(X_train, y_train)

# --- Step 4: Validate — right metric + error analysis ---
best_model = search.best_estimator_
y_pred = best_model.predict(X_test)

print(classification_report(y_test, y_pred))       # Precision/Recall/F1, not just accuracy
cm = confusion_matrix(y_test, y_pred)
print("Confusion Matrix:\n", cm)                    # inspect where errors concentrate

# Error analysis: pull out misclassified rows for manual inspection
misclassified = X_test[y_pred != y_test]
```

## Section 13 Summary

```
Tune the Process (Hyperparameters)  →  Optimize the Pipeline (Data → Features → Model → Validation)
```

- **13.1**: Parameters are *learned*; hyperparameters are *chosen* — tuning finds the hyperparameters that lead to the best learned parameters. Use Grid Search (thorough) or Randomized Search (fast) depending on budget.
- **13.5**: Optimize systematically and in order — **data quality first**, then features, then the model itself, and always finish with proper validation and error analysis. Never skip straight to model tuning if the data isn't clean.
